# V.1.3.4 Month Surface And Widget Performance Follow-Up

## Scope

This follow-up addresses three remaining V.1.3.4 behaviors:

1. Completed and cancelled tasks must not appear in month day cells in the app or Month/Multi widgets.
2. Month-widget navigation must never combine a new month header with old month content and must remove avoidable interaction latency.
3. Single-day task pills in the app month view must follow occurrence-time order instead of alphabetical order without changing multi-day placement.

Other calendar views, task lists, task widgets, and the existing priority color/glow/motion/scale behavior remain unchanged.

## Month Task Visibility

Introduce one shared month-surface visibility rule:

- A task is visible when it is not completed and its effective status is not `CANCELLED`.
- A task is hidden when `isCompleted` is true, status is `COMPLETED`, or status is `CANCELLED`, using case-insensitive status comparison.
- Month surfaces do not honor the general "show completed tasks" preference because this follow-up explicitly excludes inactive tasks from month day cells.

Apply the rule before indexing tasks by date in the app Month view and compact month overview, and before creating candidates in `WidgetMonthPageSource`. This prevents hidden tasks from consuming lanes or contributing to overflow counts.

## App Month Ordering

Extend the app month-row candidate model with an occurrence sort timestamp:

- Events use `startsAtMillis`.
- Tasks use `startAtMillis`, then `dueAtMillis`, then `Long.MAX_VALUE`.

Preserve the existing multi-day comparator exactly: longer spans first, then start date, then title. For candidates whose span is one day, compare occurrence timestamps before title. This orders single-day events and tasks together by when they occur while leaving multi-day lane, continuation, fade, and overflow behavior unchanged.

## Widget Navigation Pipeline

Remove the current header-only acknowledgement and the fixed 300 ms coalescing delay. Navigation remains revision-based and latest-wins.

Maintain a rolling model cache covering the selected month plus six months in each direction. The cache is keyed by month, widget settings, time zone, render signature, and data generation. Cache warming runs in the existing background widget scope and is cancelled or ignored when its data generation becomes stale.

For each navigation command:

1. Commit the new month and revision synchronously.
2. Cancel obsolete work for the same widget.
3. If the complete target page is cached, render and publish the complete target in one update.
4. If the target is not cached, publish a complete target-month skeleton in one update, load the target page immediately, and replace the skeleton with the complete page in one update.
5. Check the revision immediately before every widget update so an obsolete target can never overwrite a newer rapid navigation target.
6. Recenter background cache warming on the final selected month.

The skeleton includes the target month header and stable calendar geometry but no stale event/task content. It is a full replacement, not a partial update, so header and body always describe the same state.

The Month and Multi widgets use the same policy. Month switching adds no frame-based animation. Existing task priority animation behavior is not changed.

## Performance Constraints

- There is no intentional delay on the cached path.
- Database/model loading starts immediately on a cache miss.
- Rapid commands may skip intermediate visual states, but the final displayed month must equal the final committed navigation state.
- The implementation caches lightweight month page models, not complete `RemoteViews`, to avoid retaining large widget trees and bitmap payloads.
- The rolling cache capacity must accommodate at least one complete 13-month window plus normal overlap between Month and Multi widgets.

Launcher rendering and IPC scheduling remain controlled by Android, but the app will no longer add a 300 ms delay or submit a mismatched two-stage update.

## Testing

Add focused regression tests that prove:

- Completed, `COMPLETED`, and `CANCELLED` tasks are excluded from month surfaces.
- Active tasks remain visible.
- Single-day mixed event/task candidates are ordered by occurrence timestamp.
- Existing multi-day ordering remains unchanged.
- A cache hit selects a complete page with no skeleton stage.
- A cache miss selects a full skeleton followed by a complete page.
- Rapid navigation revisions only permit the latest target to apply.
- The rolling warm window contains the center month plus six months in each direction.

Run targeted unit tests first, then the full unit suite, debug assembly, connected tests, installation, and on-device Month/Multi widget checks with rapid repeated previous/next taps.

## Non-Goals

- Rewriting the widget as a single bitmap or with Jetpack Glance.
- Caching complete `RemoteViews` trees.
- Removing or redesigning task priority effects.
- Changing completed-task visibility outside month day cells.
- Changing multi-day month placement or continuation semantics.
