# V.1.3.4 Reliability Refactor Design

**Status:** Approved design for implementation planning

## Summary

V.1.3.4 will fix the reported synchronization, reminder, gesture, lifecycle, editor, and widget defects while decomposing the two largest UI/widget files into focused modules. The refactor will be incremental: characterize existing behavior, extract a boundary without behavior changes, then implement and verify the related fixes.

The release metadata will become `versionCode 14` and `versionName "V.1.3.4"`.

## Safety And Delivery Model

- Work in the isolated `codex/v1.3.4-refactor` worktree. Keep `main` untouched until integration.
- Use a fresh implementation agent for each bounded task and a separate reviewer for its exact diff.
- Keep changes to shared Compose and widget hotspots sequential. Parallel agents may perform independent analysis or review, but must not concurrently edit overlapping files.
- Add characterization tests before structural moves. Each extraction commit must preserve behavior and pass its targeted tests before a behavior-changing commit begins.
- Run Gradle tasks sequentially on JDK 17. Use the existing local Android SDK through the environment without adding or changing `local.properties`.
- Verify gesture, notification, and `RemoteViews` behavior on the attached Android device after the relevant chunks.

## Architecture

### Source And Task Mutation Coordination

`CalendarRepository` remains responsible for persistence and protocol operations and exposes one explicit full-refresh primitive. A new `SourceCalendarMutationCoordinator` owns structural source/calendar commands and serializes the follow-up full refresh, diagnostics update, reminder reconciliation, and widget update. This avoids duplicated or overlapping global refreshes from the ViewModel, foreground refresh, WorkManager, and observers.

Full refresh means `syncNow(includeDisabledProviderCalendars = currentSetting, forceFullCalDavRefresh = true)`, followed by diagnostics, reminder reconciliation, and widget refresh.

Run a full refresh after:

- adding, renaming, editing, or removing a source;
- creating, deleting, renaming/recoloring, enabling, or disabling a calendar;
- changing source credentials, username, server URL, or subscribed read-only URL.

Do not run a full refresh after calendar ordering or `visible in views` changes. Those are applied locally and immediately. The Android-provider `show disabled calendars` setting retains its targeted provider refresh rather than triggering an unrelated global refresh.

Mutation success and refresh success are separate outcomes. If a structural mutation succeeds but refreshing another source fails, the mutation remains saved and the UI reports that refresh failed instead of implying a rollback.

A shared task-mutation coordinator will route status changes from the app, task widgets, and notification actions through the same persistence, pending-change push, notification reconciliation, reminder reschedule, and widget-update path.

### Occurrence Identity, Reminders, And Launch Targets

Use durable resource identities for application operations. An iCalendar UID remains protocol metadata and is not treated as a globally unique database key.

Introduce explicit occurrence identities:

```kotlin
data class EventOccurrenceId(
    val resourceHref: String,
    val recurrenceIdMillis: Long,
)

data class TaskOccurrenceId(
    val resourceHref: String,
    val recurrenceIdMillis: Long,
)
```

Non-recurring events use their original start timestamp as `recurrenceIdMillis`. Non-recurring tasks use their start timestamp when present and otherwise their due timestamp; tasks without either timestamp cannot produce a reminder occurrence. Recurrence expansion preserves the original recurrence identifier in an occurrence envelope instead of discarding it.

Split reminder responsibilities into:

- `ReminderPlanner`: pure creation of alarm and notification plans;
- `ReminderRegistry`: persisted scheduled-alarm and posted-notification records;
- `ReminderScheduler`: Android `AlarmManager` and `NotificationManager` integration.

Every reminder alarm has an independently cancellable identity based on occurrence, trigger time, and reminder offset. All reminders for one occurrence share one stable notification tag/ID, so a later reminder replaces the older tray entry. Different occurrences, including separate occurrences of the same recurring series, may coexist.

Reminder reconciliation removes posted notifications and scheduled alarms when a task occurrence is completed or cancelled. A whole-series completion, cancellation, deletion, collection disable/delete, or source deletion clears all matching occurrence records.

A shared `CalendarLaunchTarget` is encoded in widget and reminder intents. A task reminder target carries `resourceHref`, `recurrenceIdMillis`, and occurrence date. Explicit intent actions and unique intent data define `PendingIntent` identity. `MainActivity` decodes the target and asks the ViewModel/repository to resolve the exact occurrence before opening its detail popover. Cold launch and `singleTop` delivery use the same path.

### Time And Foreground Lifecycle

`CalendarTimeState` is a lifecycle-aware app-level source for the current minute and date. It emits:

- immediately when the app enters the foreground;
- at aligned minute boundaries;
- after system time, date, or time-zone changes.

Today markers, the toolbar date icon, month/day headers, agenda grouping, and the current-time line read this shared state instead of independent `LocalDate.now()` or a composition-relative 60-second loop.

`ForegroundRecenterPolicy` observes process lifecycle rather than Activity recreation, persists the time the process entered the background, and uses an injected clock for tests. At `>=120_000` milliseconds away, it emits a dedicated recenter request. Explicit notification/widget launch targets take precedence over automatic recentering.

Recenter behavior:

- Day and multi-day views move the pager to the current day.
- Month view moves the month list to the current month and selects today.
- Agenda moves to the current date.
- Open editors, detail popovers, drawers, dialogs, and their in-progress state remain open; only the underlying calendar recenters.

### Compose Decomposition And State Ownership

Replace mutable theme globals with immutable UI tokens supplied through composition locals. Then extract leaf components before orchestrators.

Planned UI boundaries:

- `ui/editor`: canonical editor schedule state, shared editor controls, event editor, and task editor;
- `ui/timeline`: viewport state, drag state, all-day layout/viewport, timed cards, draft wireframe, day pager column, and timeline orchestrator;
- `ui/month`: month overview gesture state, month overview, grid, and strip;
- `ui/time`: live temporal state and foreground recenter request handling;
- `KgsCalendarApp` and `CalendarShell`: final thin routing/orchestration layers after leaf extraction.

`EditorScheduleState` is the single schedule source for editor fields and the timeline wireframe. It stores raw date/time field text, presence flags, and all-day state while exposing a last-valid timeline preview. This allows temporarily invalid text during editing without disconnecting the wireframe. Wireframe moves/resizes update the same state, and editor field changes derive the preview from that state without feedback loops. The state is keyed to the creation/editor session so it survives tiny/full sheet changes and event/task conversion.

`TimelineDragSession` stores item/occurrence identity, root-space pointer position, target date, target zone, and reserved all-day lane. Timed event and task cards use one shared drag implementation.

Timed-to-all-day behavior:

- The card is rendered in a root overlay so it can visually leave the clipped timed grid.
- A stable, non-animated hover boundary with hysteresis decides whether the pointer targets the all-day section.
- Entering the zone snaps the card to the candidate all-day lane and adds a transient reservation to `AllDayLayoutModel`.
- If the all-day area has no items, the reservation creates one animated lane of space.
- Leaving the zone removes the reservation and returns the card to a timed snap target.
- Releasing commits the target already shown by the preview. Cancelling restores the original item.
- Height, card position, lane changes, and reservation removal use the existing motion easing family.

`TimelineViewportState` owns hour height, scroll offset, viewport/content dimensions, one-finger scroll, and pinch lifecycle. One ancestor-level detector claims the complete gesture as soon as two pointers are active. The gesture uses cumulative span relative to its starting scale and anchors the content minute under the current finger centroid. It supports moving both fingers or keeping either finger stationary. One-finger scrolling and pager competition remain disabled until fewer than two pointers remain. Non-interactive morph copies do not report viewport dimensions or own gestures.

The month overview uses one axis-locked gesture controller scoped to the month grid. Direction locks after touch slop. Horizontal intent pages months even with minor vertical drift; vertical upward intent dismisses the overview. A new gesture interrupts any in-progress settle animation, and rapid swipes use latest state rather than waiting for an earlier animation.

### Widget Decomposition

Split widget responsibilities without replacing the classic `RemoteViews` architecture:

- `WidgetMonthModel`: Android-free month candidates, lane/layout logic, cells, pages, and segments;
- `WidgetMonthRenderSpec`: size buckets and render-capacity calculations;
- `WidgetMonthPageSource`: Room/entity-to-month-candidate loading;
- `WidgetMonthNavigation`: atomic month state and latest-wins navigation coordination;
- `WidgetMonthRemoteViewsRenderer`: root/page/cell/chip rendering and action bindings;
- `WidgetTaskCardRenderer`: canonical task-card geometry, typography, glyphs, hierarchy, and priority frames;
- focused persistent state and passive cache modules extracted from `WidgetState`.

Month navigation mutations return an atomic snapshot containing widget ID, target month, and revision. The renderer receives that snapshot rather than rereading mutable state. Every launcher update checks that its revision is still current. Rapid next/previous/today taps therefore acknowledge the newest target immediately and stale renders cannot overwrite it.

Navigation applies an immediate target-month preview, uses a cached populated page when available, then performs authoritative revalidation. Cache keys include time zone and a repository/data generation so a cache hit cannot remain stale indefinitely. The Month and Multi widget month panels use the same coordinator.

Month cell ordering adds an occurrence-time key only for single-day candidates. Existing visible-start, multi-day span, lane, continuation, and fade logic remains unchanged. Same-time single-day items use title as a stable tie-breaker.

Task cards use one canonical base specification for rects, typography, checkbox/status geometry, padding, hierarchy, chevron, and corner radius. Static and priority-animated rows rasterize the same base. Priority color, glow, shake, bounce, and scale remain enabled and transform that base; priority never selects alternate font or checkbox metrics. Transparent XML views remain as stable click targets where required by `RemoteViews`.

The renderer preserves API 26-30 `RemoteViewsService` behavior, API 31+ direct collection behavior, fixed span XML layouts, stable view IDs, pending-intent templates, and bitmap URI lifetime constraints.

## Implementation Chunks

1. Add characterization tests, immutable UI tokens, shared resource/occurrence identities, and package-visible models.
2. Extract repository refresh primitive and add source/calendar and task mutation coordinators.
3. Add reminder planner/registry, per-occurrence replacement, cancellation reconciliation, and shared launch targets.
4. Add shared temporal state and foreground recenter policy.
5. Extract editors and migrate editor/wireframe scheduling to one session state.
6. Extract timeline leaf components, add all-day drag reservation, then replace pinch ownership.
7. Extract month overview and replace competing gesture detectors with axis-locked state.
8. Extract widget model/navigation/rendering/task-card modules and implement widget fixes.
9. Extract the final shell/orchestrators, bump release metadata, run complete verification, and perform whole-branch review.

The editor/wireframe, timeline drag/reservation, pinch/viewport, temporal/recenter, and widget navigation/cache chains remain sequential. A later task may consume only interfaces approved by the preceding task review.

## Verification

### Unit Tests

- Structural mutation matrix: every add/remove/edit/enable action refreshes; ordering and view visibility do not.
- Refresh serialization and separate mutation/refresh failure outcomes.
- Exact resource and occurrence identity, including duplicate iCalendar UIDs in different collections.
- Alarm uniqueness, per-occurrence notification replacement, completed/cancelled reconciliation, recurring overrides, and whole-series cancellation.
- Launch-target encoding/decoding and exact occurrence resolution.
- Foreground durations below and at/above 120 seconds, persisted process restoration, explicit-target precedence, and aligned minute/day updates.
- Editor-field-to-wireframe and wireframe-to-editor round trips, all-day conversion, invalid raw text, snapping, and range validation.
- Root-space drag target, hover hysteresis, empty-area lane reservation, lane removal, and final drop action.
- Pinch scale/anchor calculations for two moving fingers and either stationary finger.
- Month overview touch-slop axis locking and interruption of settle animations.
- Widget rapid navigation generations, opposite taps, cache hit/miss/revalidation, and stale revision rejection.
- Single-day widget occurrence ordering, equal-time ties, and unchanged mixed/multi-day lane behavior.
- Identical task base geometry for priorities `null` and `1..9`, with priority differences confined to effect state.

### Connected And Device Tests

- Compose tests for editor/wireframe synchronization, all-day reservation, pinch ownership, rapid month gestures, temporal refresh, and recentering beneath an open overlay.
- Notification cold-launch and `singleTop` deep links, exact recurring occurrence, replacement, and dismissal from app/widget/action status changes.
- Launcher checks for rapid Month and Multi widget navigation, resize behavior, task-card parity, priority animation, and day-cell ordering.
- Manual touch checks for dragging timed cards into and out of empty/populated all-day sections and for all supported pinch finger patterns.

### Build Gates

After each chunk, run targeted tests, `:app:testDebugUnitTest`, and `:app:assembleDebug` sequentially. Run relevant connected tests and device checks after gesture, notification, lifecycle, and widget chunks. The final gate runs the complete unit and connected suites, builds the debug APK, executes the full reported-bug checklist, performs an independent whole-branch review, and confirms no SDK path, signing file, secret, generated APK, heap dump, or build artifact is tracked.

## Assumptions

- The two-minute threshold is fixed at 120 seconds for V.1.3.4 and is not exposed as a setting.
- Foreground recentering never dismisses or recreates an open overlay.
- Notification replacement is scoped to one item occurrence, not an entire recurring series.
- Calendar ordering and view visibility remain local-only operations.
- Existing priority animation appearance remains; only the base card inconsistencies are removed.
- Existing multi-day month-widget layout semantics remain unchanged.
- V.1.3.4 introduces no Room schema migration. Recurrence identity is carried by runtime occurrence envelopes, launch targets, and the reminder registry rather than new entity columns.
