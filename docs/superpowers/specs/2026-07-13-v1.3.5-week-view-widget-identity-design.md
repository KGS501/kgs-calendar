# V.1.3.5 Week View and Widget Event Identity Design

## Summary

V.1.3.5 adds an optional, week-aligned form of the existing Multiple days timeline and fixes the Month widget joining independent events that share a title and color. Existing users retain Multiple days behavior until they enable Week view.

The implementation keeps `CalendarViewMode.ThreeDay` as the persisted navigation slot. Settings determine whether that slot is presented as Multiple days or Week, which avoids a selected-view migration and preserves the established timeline, all-day, drag-and-drop, pinch, and day-morph code paths.

## Goals

- Let users replace Multiple days with a seven-day Week view.
- Align Week view to the configured first day of week whenever it is entered or explicitly navigated.
- Let Week view either snap by one complete week or continue day by day.
- Preserve tapping a timeline date header to switch between 1-day and Multiple days/Week.
- Update navigation and settings labels to match the selected timeline type.
- Keep independent same-name events separate in the Month widget.
- Preserve genuine multi-day event layout, continuation, row-wrap, and fade behavior.
- Advance release metadata to version name `V.1.3.5` and version code `15`.

## Non-Goals

- Adding Week as a second persisted `CalendarViewMode` enum value.
- Displaying Multiple days and Week as separate navigation destinations at the same time.
- Rewriting the timeline as a separate week-only pager.
- Changing the in-app Month view's existing multi-day layout.
- Changing how event or task identity is stored in Room or synchronized from providers.

## Settings Model

Add two persisted settings:

- `weekViewEnabled`, a Boolean that defaults to `false`.
- `fullWeekSwipeEnabled`, a Boolean that defaults to `true`.

The defaults preserve existing installations while making full-week paging the initial behavior when a user first enables Week view. Disabling Week view does not erase either the configured Multiple days count or the full-week swipe preference.

Expose both settings through `SettingsStore`, `CalendarViewModel`, and `CalendarUiState`. Keep the internal selected view as `CalendarViewMode.ThreeDay` for both Multiple days and Week.

## Settings Interface

Under **Settings > Behavior > View**, place the following controls after **First day of week**:

1. **Use week view**
   - Subtitle: **Shows seven days aligned to your first day of week.**
2. **Swipe full weeks**
   - Subtitle: **Moves one whole week per horizontal swipe. Turn off to scroll by individual days.**

`Swipe full weeks` appears with the existing fade-and-expand settings animation only while Week view is enabled. The existing Multiple days side-button control and fixed day-count slider are hidden with the complementary animation while Week view is enabled. Their stored values remain unchanged.

Add equivalent German strings for every new label and subtitle.

## Dynamic Labels

When `weekViewEnabled` is `false`, the timeline destination remains labeled **Multiple days**. When it is `true`, the same destination is labeled **Week**.

Apply the dynamic label in:

- The navigation drawer.
- The Default view value shown in settings.
- The Default view selection dialog.

Other view labels and internal `CalendarViewMode` names remain unchanged.

## Timeline Window Policy

Introduce a small, pure timeline-window policy that derives visible-day behavior from:

- Whether the timeline is in 1-day or `ThreeDay` mode.
- `weekViewEnabled`.
- `fullWeekSwipeEnabled`.
- `multiDayCount`.
- `firstDayOfWeek`.

The policy is responsible for:

- Returning a day count of one for Day, seven for Week, or the configured count for Multiple days.
- Finding the start of the week containing a date with `previousOrSame(firstDayOfWeek)` semantics.
- Choosing the visible anchor date when entering Week view.
- Calculating toolbar previous/next targets.
- Calculating week-aligned horizontal swipe targets.

Keeping these rules pure makes date behavior independently unit-testable and prevents pager details from leaking into settings or repository code.

## Week Entry and 1-Day Morph

Tapping a date header continues switching between Day and the configured `ThreeDay` presentation.

When expanding from Multiple days or Week into Day, the tapped date remains the focused date and grows from its current column, preserving the existing morph.

When returning from Day to Multiple days, the focused day remains the first visible day, matching current behavior.

When returning from Day to Week:

- Compute the configured week start containing the focused date.
- Render the seven dates from that start through six days later.
- Place the focused date in its actual weekday column during the morph.
- Hand the settled pager off to the computed week-start anchor instead of moving the focused date to the left edge.

The morph context must therefore distinguish the focused date from the target visible anchor. This prevents a Wednesday, for example, from becoming the leftmost day when a Monday-start week is restored.

## Horizontal Navigation

The existing day-page `HorizontalPager` remains the rendering foundation. Week behavior is added through policy rather than a second pager implementation.

### Full-Week Paging

- The pager starts on a page aligned to `firstDayOfWeek`.
- A completed horizontal swipe settles exactly seven days backward or forward.
- Rapid or high-velocity gestures still move one complete week per gesture.
- The settled page remains aligned to the configured week start.

Use a custom `PagerSnapDistance` backed by a pure target-page helper. The helper treats an unchanged suggested target as no navigation and otherwise chooses the previous or next aligned week page based on gesture direction.

### Seamless Paging

- Entering Week still aligns the first seven-day window to `firstDayOfWeek`.
- After entry, the existing day-page fling behavior remains active.
- Horizontal gestures may settle on any day, allowing the seven-day window to move continuously.
- After a day-level settle, that day becomes the new leftmost visible anchor; it is not immediately realigned to `firstDayOfWeek`.

### Toolbar Navigation

Previous and next period actions move seven days in Week view regardless of swipe mode. Multiple days retains its configured-count movement, while Day, Month, Agenda, and Tasks retain current behavior.

Changing `firstDayOfWeek` while Week view is active recomputes the containing week anchor and updates the visible/data range without requiring an app restart.

## Data Ranges

Week view loads seven days beginning at its current visible anchor. That anchor is the computed week start on entry and during full-week paging; after a seamless day-level settle it is the settled day. The broader timeline data buffer should use the same anchor so events and tasks are present throughout the visible window and during adjacent-page gestures.

Multiple days continues using its configured count and existing buffered range. Day and all non-timeline views retain their current range behavior.

## Month Widget Root Cause

`WidgetMonthModel.layout` currently infers continuity between adjacent candidates when they occupy the same lane and have matching normalized titles and colors. It then sets continuation flags even when their candidate IDs differ. `monthWeekSegments` consumes those flags and renders one connected span.

This heuristic incorrectly joins independent events such as two separate timed events named **Swim Practice** on consecutive days. True multi-day occurrences already enter the model as one `WidgetMonthCandidate` with one occurrence ID and an actual start/end date range, so title/color inference is not required.

## Month Widget Fix

- Derive continuation flags only from a candidate's actual start and end dates.
- Remove title/color-based adjacency inference between distinct candidates.
- Require matching occurrence IDs when `WidgetMonthItem.visuallyContinuesInto` joins adjacent day cells.
- Retain the current lane allocator, occurrence-time ordering, multi-day priority, continuation fades, and week-row wrapping.

The existing `monthOccurrenceKey`, which includes resource/UID identity and occurrence start time, remains the event candidate identity source.

## Testing Strategy

Follow test-driven development for each behavior.

### Timeline Policy Tests

- Week anchors correctly for each possible `DayOfWeek` start.
- Week entry contains and correctly positions the previously focused date.
- Full-week swipe targets move exactly seven aligned pages in either direction.
- An insufficient gesture that suggests no page change remains on the current week.
- Seamless mode retains day-level target pages after its initial aligned entry.
- Toolbar navigation moves seven days in Week and the configured count in Multiple days.
- Visible ranges use the week anchor and seven-day extent.

### Settings Tests

- Week view defaults off.
- Full-week swipe defaults on.
- Stored values round-trip through DataStore.
- Multiple days values survive enabling and disabling Week view.
- Dynamic labels select Week or Multiple days correctly.
- Settings visibility policy shows only controls relevant to the active presentation.

### Widget Regression Tests

- Two single-day candidates with different IDs but identical titles, colors, and times on consecutive dates remain two one-day segments.
- Adjacent same-name events in the same lane do not receive continuation flags.
- One genuine multi-day candidate remains connected across all covered dates.
- Genuine multi-day candidates still wrap and show a title at each new widget row.
- Bottom fade behavior remains correct for continued multi-day items.

### Verification

- Run focused unit tests during each red-green-refactor cycle.
- Run `.\gradlew.bat :app:testDebugUnitTest` sequentially on JDK 17.
- Run `.\gradlew.bat :app:assembleDebug`.
- Install the debug APK on the connected device when available.
- Manually verify Week/Day morphing, both swipe modes, preferred-week-start changes, and Month widget rendering because launcher behavior cannot be fully established by JVM tests.

## Release Metadata

Set the Android version name to `V.1.3.5` and version code to `15`. Building or publishing release artifacts is outside this implementation unless requested separately.
